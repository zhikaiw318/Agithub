###### **一、小修改（直接操作 main 分支）—— 核心：先拉后推，极简高效**

**适用场景**

仅修改 1-2 行代码、修复拼写错误 / 小 bug、调整注释等低风险小改动。

**完整流程（代码 + 操作一步整合）**

\# 1. 切回main分支（确保操作目标正确）

**git checkout main**

\# 2. 拉取远程main最新代码（同步同事修改，避免冲突）

**git pull origin main**    当前在 main + 已跟踪 origin/main时，*可**简写为git pull***

\# 3. 本地修改/添加文件（比如修改test.c、新增小文件）

\# （此处为示例操作，实际替换为你的修改即可）

echo '#include <stdio.h> int main() { printf("修复小bug\\\\n"); return 0; }' > test.c

\# 4. 将修改加入暂存区

**git add .**

\# 5. 提交到本地main分支（提交说明简洁清晰）

**git commit -m "fix: 修复test.c拼写错误"**

\# 6. 再次拉取（防同事在你提交期间修改，双重保险）

**git pull origin main**     *可简写为**git pull***

\# 7. 推送到远程main分支（第一次推送绑定上游，后续直接git push）

**git push -u origin main**    已用过 -u 一次（已跟踪），可简写为**git push**



**关键说明**

全程无需新建分支，直接在 main 分支操作；

核心逻辑是「先同步远程→本地修改→再同步→最后推送」，避免覆盖同事代码；

若拉取时出现冲突，解决冲突（删除 <<<<<<< 等标记）后执行git add 冲突文件 \&\& git commit，再推送即可。

###### **二、大修改 / 新功能（新建功能分支）—— 核心：隔离开发，定期同步**

**适用场景**

新功能开发、代码重构、实验性修改、多文件大改动等，避免污染 main 稳定分支。

**完整流程（代码 + 操作一步整合）**

\# 1. 切回main分支并拉取最新代码（确保分支基于最新版本）

git checkout main \&\& git pull origin main

\# 2. 新建并切换到功能分支（命名规范：类型-功能名，如feat-login、fix-pay-bug）

git checkout -b feat-new-function

\# 3. 分支内开发（修改/添加文件，可多次提交）

echo '#include <stdio.h> void new\_func() { printf("新功能开发\\\\n"); }' > new\_func.c

git add .

git commit -m "feat: 新增new\_func函数，实现XXX核心逻辑"

\# 4. 定期同步main分支（避免最终合并时冲突过多，建议每天同步1次）

git checkout main \&\& git pull origin main

git checkout feat-new-function \&\& git merge main

\# （若合并冲突，解决后执行：git add . \&\& git commit -m "merge: 同步main最新代码"）

\# 5. 功能开发完成，合并回main分支

git checkout main \&\& git pull origin main

git merge feat-new-function

\# 6. 推送到远程main分支

git push -u origin main

\# 7. （可选）删除本地功能分支（开发完成后清理）

git branch -d feat-new-function

**关键说明**

分支命名建议遵循「类型 - 功能名」，团队可快速识别分支用途；

定期同步 main 是核心：将大冲突拆分为小冲突逐步解决，降低合并难度；

功能分支仅在本地 / 远程临时存在，合并完成后可删除，保持仓库整洁；

若团队用 PR/MR（Pull Request）协作，可将git merge替换为推送分支后在 GitHub 提 PR，审核通过后合并。



方式	                                         核心流程	                         核心优势	                                     适用场景

直接操作 main	                   拉→改→提→拉→推	                 步骤少、效率高                 	     小修改 / 紧急修复

新建功能分支	           建分支→开发→同步→合并→推	          隔离开发、冲突可控、易回滚	     新功能 / 大修改



**两种方式的核心原则一致：始终保证本地代码基于远程最新版本，避免直接推送覆盖他人修改；**

**企业协作中优先用「新建分支」模式，main 分支仅保留可运行的稳定代码。**
